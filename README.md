## Project Overview

PrepArena is a full-stack, AI-powered platform designed to help software engineers and computer science students rigorously prepare for technical interviews. It combines large language model (LLM) question generation, webcam-based exam proctoring, and an interactive interview simulation environment into a single, cohesive web application.

The platform supports two preparation modes:

| Mode | Description |
|---|---|
| **Mock Test** | Timed, proctored question-answer sessions with MCQ and open-ended text/code questions generated by an LLM. Supports both generic topic-based tests and personalized tests built from the user's own resume and job description. |
| **AI Interview** | Conversational interview simulation. Current version uses a structured 6-question scripted flow. The upcoming version will use real-time voice recognition and dynamic LLM-driven follow-up questions. |

All sessions are stored locally, building a personal history that feeds into the upcoming analytics dashboard, which will visualize progress, strengths, weaknesses, and study streaks over time.

---

## Key Features

- **LLM-Generated Question Sets** — Groq API (model: `openai/gpt-oss-120b`) generates mixed MCQ + text/code questions tailored to topic, difficulty, and count.
- **Resume + JD Personalization** — Upload your resume (PDF/TXT) and job description to generate custom questions that target the gap between your experience and the role requirements.
- **Real-Time Webcam Proctoring** — OpenCV Haar cascade–based face detection runs in a background thread on the server; JPEG frames are pushed from the browser at 2 fps during tests.
- **Cheating Detection** — Flags face-absent frames, gaze-away frames, and multi-face frames; computes a cheating probability (optionally using an XGBoost model).
- **Intelligent Scoring** — Open-ended text answers are scored using keyword matching, substring matching, and token overlap ratio (Jaccard-style).
- **Keyword Extraction** — An in-browser keyword extractor identifies up to 30 technical skills from uploaded documents using a curated term list and capitalized-phrase heuristics.
- **Session History** — All completed tests and interviews are persisted in `localStorage`, enabling streak calculations, score tracking, and upcoming analytics.
- **Study Scheduler** — A calendar widget on the dashboard lets users schedule upcoming mock tests and interviews.
- **JWT Authentication** — Stateless authentication using `djangorestframework-simplejwt`.
- **ML Data Collection** — A standalone face detection script (`smartFaceDetector.py`) captures per-frame proctoring data to CSV, enabling future model training.

### Architecture Decisions

| Decision | Rationale |
|---|---|
| SQLite for development | Zero-config, portable. Production migration to PostgreSQL is straightforward. |
| JWT (stateless auth) | Decoupled frontend/backend; no server-side session storage required. |
| Groq API for LLM | Low-latency inference via Groq's LPU hardware; dual-key failover for reliability. |
| Push-based proctoring (no server webcam) | Browser captures frames; server never accesses the camera directly. Avoids OS-level permission complexity and enables horizontal scaling. |
| Thread-per-session proctoring | Allows concurrent test sessions without blocking HTTP request handlers. Queue-based frame dropping under load prevents memory pressure. |
| `localStorage` for session history | No backend model required for personal history in the current version; enables offline capability. |
| pdfjs-dist for PDF parsing | Client-side PDF text extraction; no file upload to server for resume/JD content. |

## Module Breakdown

### Backend (Django REST API)

#### `config/settings.py`

Central Django configuration. Key highlights:

- **JWT authentication** via `rest_framework_simplejwt.authentication.JWTAuthentication` as the default DRF auth class.
- **CORS** is fully open (`CORS_ALLOW_ALL_ORIGINS = True`) for development. This should be restricted to the frontend origin in production.
- **Environment variables** loaded via `python-dotenv`: `GROQ_API_KEY` (primary) and `GROQ_API_KEY2` (fallback).
- **Database**: SQLite3 at `BASE_DIR/db.sqlite3`. No custom models exist yet; session data lives in the browser's `localStorage`.

#### `config/urls.py` — Root URL Configuration

| URL Pattern | View / Handler | Auth Required |
|---|---|---|
| `api/register/` | `register_user` | No |
| `api/login/` | `TokenObtainPairView` (simplejwt) | No |
| `api/refresh/` | `TokenRefreshView` (simplejwt) | No |
| `api/generate-mock-test/` | `generate_mock_test` | No (open in dev) |
| `api/proctoring/start/` | `start_proctoring` | No (open in dev) |
| `api/proctoring/frame/` | `push_proctoring_frame` | No (open in dev) |
| `api/proctoring/stop/` | `stop_proctoring` | No (open in dev) |
| `admin/` | Django Admin | Superuser |

#### `api/views.py` — View Functions

**`generate_mock_test(request)` — `POST /api/generate-mock-test/`**

Generates a structured question set using the Groq LLM API with automatic dual-key failover.

- **Standard mode**: prompt specifies topic, difficulty, and count; requires ~70% MCQ and ~30% open-ended questions.
- **Custom mode**: prompt instructs the model to analyze the delta between the candidate's resume skills and the job description requirements, generating targeted questions for gap areas.
- Response schema per question:
  - MCQ: `{type: "mcq", q, options: [4 strings], answer: 0–3}`
  - Text/Code: `{type: "text"|"code", q, expected_answer, expected_keywords: [4–10 strings]}`
- Robust JSON extraction: strips markdown fences, uses regex to isolate the JSON array, validates all fields before returning.

**`register_user(request)` — `POST /api/register/`**

Creates a Django `User` from `{username, email, password}`. Returns `201` on success or `400` if username already exists.

**`start_proctoring(request)` — `POST /api/proctoring/start/`**

Initializes a new proctoring session in the backend. Returns `{"session_id": "<uuid>"}`.

**`push_proctoring_frame(request)` — `POST /api/proctoring/frame/`**

Accepts a raw JPEG frame (multipart) and enqueues it for analysis. Non-blocking — frames are dropped if the queue is full to prevent memory pressure.

**`stop_proctoring(request)` — `POST /api/proctoring/stop/`**

Drains the frame queue, computes the cheating verdict, cleans up the session, and returns the full proctoring report.

#### `api/proctor.py` — Proctoring Engine

The proctoring engine is the most technically complex component of the backend. It operates entirely server-side and never accesses the webcam directly; the browser captures frames and pushes them via HTTP.

**Session lifecycle:**

```
start_session()
    └── Creates _Session(uuid), starts background worker thread
         └── _run() loop: dequeue → cv2.imdecode → _analyse_frame()

push_frame(sid, jpeg_bytes)
    └── Non-blocking enqueue to session's queue (maxsize=30)
         └── Drops frame if queue full (load shedding)

stop_session(sid)
    └── Joins worker thread → result() → removes session
```

**`_analyse_frame(frame_bgr, tracked)` — Core Analysis:**

1. Downscale to max 320px width for performance.
2. Detect frontal faces via `detectMultiScale` (scaleFactor 1.1, minNeighbors 3).
3. Detect profile faces left and right (via horizontal flip); deduplicate by IoU < 0.3.
4. Filter each candidate: aspect ratio (0.6–1.2) + HSV skin color (≥20% skin pixels).
5. Temporal tracking: require 2 consecutive frames to confirm a face (reduces false positives from hands, books, etc.).
6. Gaze analysis: detect eyes in upper half of face ROI; compute pupil centroid via adaptive threshold + contour moments; classify as `center`, `left`, `right`, `up`, or `down`. "Looking away" = no eyes found or gaze not centered.
7. Streak filtering: require 4 consecutive away-frames before incrementing `looking_away_frames` (suppresses blink/glance noise).

**Cheating verdict thresholds (fallback):**

| Metric | Flag Threshold |
|---|---|
| Face absent | ≥ 15% of frames |
| Looking away | ≥ 25% of frames |
| Multiple faces | ≥ 5% of frames |

If `proctor_model.pkl` (XGBoost) is present, it overrides the threshold-based verdict using `predict_proba ≥ 0.65`.

**Verdict response shape:**

```json
{
  "cheating": false,
  "camera_available": true,
  "face_not_in_frame_pct": 3.2,
  "not_looking_pct": 8.1,
  "multi_face_pct": 0.0,
  "total_frames": 186,
  "flags": [],
  "cheating_probability": 0.12
}
```

#### `pages/Dashboard.jsx` — Main Hub

The dashboard is the central hub of the application. It integrates:

- **Document Upload** (resume + JD): drag-and-drop or file picker. Parses via `extractTextFromFile`, computes keywords via `extractKeywords`, stores in `PrepContext`.
- **Quick Stats**: Sessions Done, Avg Score, Day Streak, Best Score — all computed from `sessions` array.
- **Session History List**: Displays last N sessions with type, topic, score, and date badges.
- **Study Scheduler**: A Monday-first calendar grid. Users add `{date, type (mock/interview/both), topic}` entries. Scheduled days are highlighted with color-coded badges.
- **Navigation**: Cards for "Start Mock Test" and "Start AI Interview" (with document status indicators).

**`computeStreak(sessions)`**: Traverses session history sorted by `dayKey` in descending order, counting consecutive calendar days with at least one session. Used for the "Day Streak" stat.

#### `pages/MockTestSetup.jsx` — Test Configuration

Two modes, toggled via a tab selector:

| Mode | Required Inputs | Available Topics |
|---|---|---|
| **Standard** | Topic, Difficulty (Easy/Medium/Hard), Count (5/10/15/20) | 10 CS domains |
| **Custom** | Resume or JD uploaded in Dashboard, CustomDiff, CustomCount (12/15/20) | Auto-derived from resume+JD |

Estimated test duration (`count × 1.5 min`) is shown as a summary pill. Shows document readiness cards (green ✓ / red ✗) for the Custom mode, preventing start if no documents are uploaded.

#### `pages/MockTest.jsx` — Active Test Session

The most complex frontend component. Runs through three logical phases:

**Phase 1: Camera Check (`cam-check`)**

- Requests `navigator.mediaDevices.getUserMedia({video: true})`.
- Displays live preview via `checkVideoRef`.
- Three paths: "Begin Test" (with camera + proctoring), "Begin Without Camera" (no proctoring), "Skip proctoring" (same as without camera).

**Phase 2: Test (`test`)**

Layout:
- Top: progress bar + question counter + timer + question navigator dots.
- Center: question text + MCQ options or textarea for text/code answers.
- Bottom: Previous / Next / Submit buttons. Floating camera feed (bottom-right).

**Timer**: Starts at `questions.length × 90` seconds. Auto-submits at 0.

**Keyboard shortcuts**:
- Number keys `1–4`: select MCQ option.
- Arrow keys `←/→`: previous/next question.
- `Enter`: advance to next question.

**Frame capture loop**: After test start, initializes a `setTimeout` chain. Each iteration:
1. Draws the live video to an off-screen canvas (480×360 max).
2. Calls `canvas.toBlob('image/jpeg', 0.7)`.
3. `POST /api/proctoring/frame/` as multipart.
4. Schedules next frame 500ms after POST resolves (not fixed interval — prevents queue backup under slow connections).

**Scoring (`handleFinish`)**:
- MCQ: 1 point if selected option index === `question.answer`, else 0.
- Text/code: `scoreTextAnswer(question, userText)` — 1.0 / 0.5 / 0.0 based on keyword ratio and token overlap.

**`scoreTextAnswer` algorithm**:
1. Normalize both texts (lowercase, collapsed whitespace).
2. Compute `keywordRatio = matchedKeywords / totalKeywords`.
3. Compute `tokenOverlapRatio` (Jaccard-style: `|intersection| / max(|A|, |B|)`).
4. Full credit (1.0): exact match, or answer contains expected (>12 chars), or kw ratio ≥ 0.8, or (kw ratio ≥ 0.7 and overlap ≥ 0.6).
5. Partial (0.5): kw ratio ≥ 0.55 or overlap ≥ 0.6.
6. No credit (0.0): otherwise.

#### `pages/MockTestResults.jsx` — Results & Verdict

Displays after test completion:

- **Score ring**: SVG `stroke-dasharray` / `stroke-dashoffset` animated ring. Green if ≥ 60%, yellow if ≥ 40%, red otherwise.
- **Proctoring card**: conditional on whether camera was available. Shows three percentage stats. Lists any flags. Uses `ShieldCheck` (clean), `ShieldAlert` (flagged), or `ShieldOff` (no camera) icons.
- **Question-by-question breakdown**: collapsible list showing Correct / Partial / Wrong badge, user answer vs expected answer, and matched keywords for text questions.
- **Session persistence**: calls `addSession()` once on mount (guarded by `savedRef`) to write to `localStorage`.

#### `pages/AIInterview.jsx` — Interview Simulation (Current)

A 6-question scripted chat interface:

| Q# | Domain |
|---|---|
| 1 | Introduction and background |
| 2 | Technical — Process vs Thread |
| 3 | System Design — URL Shortener |
| 4 | Behavioral — Debugging production issue |
| 5 | Behavioral — Technical disagreements |
| 6 | Reverse — Candidate questions |

Questions interpolate `{company}` and `{role}` from the setup page if provided. AI responses are simulated with 1200–2000ms delays. Session recorded as `{type: 'AI Interview', pct: null, status: 'Reviewed'}` when complete.

> **Note**: The scripted interview will be entirely replaced by the upcoming Real-Time Voice AI Interview system.

#### `utils/extractKeywords.js`

Two-pass keyword extraction:

1. **Curated matching** — 100+ `KNOWN_TERMS` list covering: languages (Python, JavaScript, Java, C++, Go, Rust, SQL, Bash, TypeScript), frameworks (React, Django, FastAPI, Spring Boot, Express, Vue, Angular), ML/AI (PyTorch, TensorFlow, LLM, Transformers, scikit-learn), cloud (AWS, Azure, GCP, Docker, Kubernetes, Terraform, CI/CD), databases (PostgreSQL, MongoDB, Elasticsearch, Snowflake), CS fundamentals (Data Structures, Algorithms, System Design, Distributed Systems), and methodologies (Agile, Scrum, DevOps).
2. **Heuristic extraction** — regex `\b([A-Z][a-zA-Z]+(?:\s[A-Z][a-zA-Z]+)?)\b` for uncovered capitalized tool names, filtered through a ~60-word stopword list.

Returns up to 30 unique keywords.

#### `utils/extractText.js`

Client-side document parser:

- **PDF** (`.pdf`): `pdfjs-dist` — `file.arrayBuffer()` → `pdfjsLib.getDocument()` → all pages → `page.getTextContent()` → joined text.
- **Plain text** (`.txt`, `.doc`, etc.): `FileReader.readAsText(file)` — synchronous read returning text string.

No files are uploaded to the server during document parsing. All extraction happens in the browser.

### ML / Proctoring Engine

#### `ML/FaceInFrame/Smart-Face-Detector/smartFaceDetector.py`

Standalone tool that opens the system webcam for live proctoring, designed for ML training data collection.

**Purpose**: Generate labeled CSV data to train the XGBoost `proctor_model.pkl` used by the backend's `proctor.py`.

**Detection pipeline** (mirrors `proctor.py`):
1. Detect frontal + profile faces → filter by aspect ratio (0.6–1.2) and skin color (≥30% skin pixels).
2. Temporal smoothing: require 3 consecutive frames to confirm a face.
3. Per-eye gaze detection: adaptive threshold → Gaussian blur → contour moments for pupil centroid. Classify `left`/`right`/`up`/`down`/`center` using tighter thresholds (0.38–0.62 Hz, 0.35–0.65 Vt) than the production `proctor.py`.
4. Majority vote across eyes for session-level gaze verdict.

**Session-level cheating thresholds**:
- Face missing ≥ 15%
- Looking away ≥ 20%
- Multiple faces ≥ 5%

**Output files**:

`exam_session_data.csv` — per-frame row:

| Column | Description |
|---|---|
| `session_id` | Auto-incremented integer |
| `frame_number` | Frame index within session |
| `face_count` | Number of confirmed faces |
| `face_detected` | 0 or 1 |
| `eyes_detected` | 0 or 1 |
| `gaze_horizontal` | `left`/`right`/`center`/`away`/`no_face` |
| `gaze_vertical` | `up`/`down`/`center`/`away`/`no_face` |
| `looking_away` | 0 or 1 |
| `multiple_faces` | 0 or 1 |
| `cheating` | 0 or 1 (per-frame label) |
| `face_not_in_frame_pct` | Running session-level % |
| `not_looking_at_screen_pct` | Running session-level % |
| `multiple_faces_pct` | Running session-level % |

`session_summary.csv` — one row per session with aggregate stats and final binary `cheating_label`.

`proctoring_binary_model.ipynb` — Jupyter notebook for training the XGBoost classifier on the collected data.

### Proctoring Workflow

```
Browser                              Backend (proctor.py)
────────────────────────────────     ─────────────────────────────────────────
POST /api/proctoring/start/     →    start_session()
                                         └── uuid = str(uuid4())
                                         └── _sessions[uuid] = _Session(uuid)
                                         └── _Session._worker.start()
←  {"session_id": uuid}

[Every 500ms during test:]
canvas.toBlob(jpeg)
POST /api/proctoring/frame/     →    push_frame(sid, jpeg_bytes)
  {session_id, frame (file)}             └── session._q.put_nowait(jpeg_bytes)
←  {"ok": true}                              (drops if queue full — non-blocking)

                                     Background thread (_run):
                                         └── while True: dequeue jpeg
                                         └── cv2.imdecode(jpeg)
                                         └── _analyse_frame(frame, tracked)
                                               ├── Haar cascade: frontal + profile
                                               ├── Skin + aspect-ratio filter
                                               ├── IoU tracking (2-frame confirm)
                                               ├── Eye detection + gaze
                                               └── 4-frame streak for "away"

POST /api/proctoring/stop/      →    stop_session(sid)
  {"session_id": uuid}                   └── Join worker thread
                                         └── result() → compute percentages
                                         └── XGBoost or threshold-based verdict
←  {cheating, flags, pcts, ...}          └── del _sessions[uid]
```

## Functional Requirements

### FR-01: User Registration and Authentication
- The system shall allow users to register with a username, email address, and password.
- The system shall authenticate users using JWT tokens (access + refresh).
- The system shall allow users to log out by clearing stored tokens.
- Passwords shall be validated per Django's built-in password rules (minimum length, common password check, numeric-only check).

### FR-02: Document Upload and Parsing
- The system shall accept resume and job description files in PDF and plain-text formats.
- The system shall extract raw text from PDF files entirely client-side using pdfjs-dist.
- The system shall extract up to 30 technical keywords from combined resume + JD text.
- Extracted text and keywords shall persist in session state for use across pages.

### FR-03: Mock Test Generation
- The system shall generate question sets using a large language model (Groq API).
- Standard mode shall generate questions for a chosen topic, difficulty level (Easy/Medium/Hard), and count (5/10/15/20).
- Custom mode shall generate personalized questions based on the gap between the user's resume and the target job description.
- Generated question sets shall contain approximately 70% MCQ and 30% text/code questions.
- The system shall validate all generated questions before returning them to the client.

### FR-04: Proctored Test Execution
- The system shall request webcam access before starting a test.
- The system shall allow tests to proceed without a camera (no proctoring).
- During a test, the system shall capture and transmit JPEG frames to the backend at approximately 2 fps.
- The backend shall analyse frames for face presence, gaze direction, and multiple faces.
- The system shall enforce a per-question time budget (90 seconds per question).
- The system shall automatically submit the test when the timer expires.

### FR-05: Answer Evaluation
- MCQ answers shall be evaluated by comparing the selected index to the correct answer index.
- Open-ended text and code answers shall be evaluated using a hybrid scoring algorithm:
  - Keyword ratio (matched expected keywords / total expected keywords)
  - Token overlap ratio (Jaccard-style token set comparison)
  - Exact and substring match checks
- Scores shall be 0.0, 0.5, or 1.0 per question.

### FR-06: Proctoring Report
- On test submission, the backend shall return a proctoring verdict including:
  - Binary cheating flag
  - Face-absent percentage, gaze-away percentage, multi-face percentage
  - List of human-readable flags
  - Cheating probability (if XGBoost model is loaded)
- The results page shall prominently display the proctoring verdict alongside the test score.

### FR-07: Session History
- The system shall persist session records (type, topic, score, date) to browser localStorage.
- Session history shall support a maximum of 100 records.
- The dashboard shall compute and display: total sessions, average score, current day streak, and best score.

### FR-08: Study Scheduler
- The system shall allow users to schedule future mock tests or AI interviews on a calendar.
- The calendar shall display scheduled events with type badges.
- Users shall be able to add and remove schedule entries.

### FR-09: AI Interview (Current Scripted Version)
- The system shall present a structured 6-question interview covering technical and behavioral domains.
- Questions shall be interpolated with company name and role if provided.
- The session shall be recorded in history upon completion.

### FR-10: ML Data Collection
- The standalone `smartFaceDetector.py` tool shall collect per-frame proctoring data to CSV.
- Collected data shall include gaze direction, face count, and binary cheating labels.
- The tool shall support training an XGBoost binary classifier from the collected data.

## Non-Functional Requirements

### NFR-01: Performance
- Mock test question generation (LLM call) shall complete within 10 seconds under normal network conditions.
- Frame analysis latency shall not block HTTP request handling (background thread processing).
- The frame queue shall have a maximum capacity of 30 frames; excess frames shall be dropped gracefully to prevent memory pressure.
- The frontend shall capture and transmit frames on a chained 500ms schedule (next frame after POST resolves), preventing queue backup under slow connections.

### NFR-02: Scalability
- The proctoring engine uses one thread per session. Under high concurrency, thread pool management and horizontal scaling behind a load balancer shall be considered.
- Session state is held in-process memory; migration to Redis or a persistent store is required for multi-instance deployments.

### NFR-03: Reliability
- The LLM integration shall implement automatic failover across two API keys (`GROQ_API_KEY`, `GROQ_API_KEY2`).
- Frame pushing is non-blocking; the test session continues uninterrupted even if individual frames are dropped.
- JSON extraction from LLM responses is resilient: strips markdown fences, uses regex isolation, validates all fields.

### NFR-04: Security
- Passwords shall be stored as PBKDF2 hashes via Django's `create_user`.
- API authentication is performed via JWT Bearer tokens.
- Resume and JD text is processed entirely in the browser — raw document bytes are never transmitted to the server.
- `SECRET_KEY` and API keys shall be stored in environment variables (`.env`), never hardcoded in production.
- `CORS_ALLOW_ALL_ORIGINS` shall be restricted to the frontend origin in production.
- `DEBUG` shall be set to `False` in production.

### NFR-05: Usability
- The test interface shall support full keyboard navigation (number keys for MCQ, arrow keys for navigation, Enter to advance).
- The application shall provide immediate visual feedback on camera access state (requesting / granted / denied).
- Document upload shall support both drag-and-drop and traditional file picker interactions.

### NFR-06: Maintainability
- Frontend state is centralized in `PrepContext`, avoiding prop drilling.
- Backend proctoring logic is fully encapsulated in `proctor.py`, with a clean three-function public API (`start_session`, `push_frame`, `stop_session`).
- LLM prompt construction and JSON validation are separated from HTTP handler logic.

### NFR-07: Offline Capability
- Session history, uploaded document text, and keywords are stored in browser memory/localStorage.
- The dashboard's stats, streak, and scheduler function without internet connectivity once documents are loaded.

## Roadmap — Upcoming Features

### Real-Time Voice AI Interview

The current scripted AI interview module will be replaced with a fully dynamic, voice-enabled interview system. Below is the planned architecture.

#### Overview

```
User speaks → browser STT → transcript → LLM (Groq) → dynamic next question → TTS → spoken response
```

#### Core Components

**1. Voice Input — Speech-to-Text (STT)**

- Browser-native: `window.SpeechRecognition` / `webkitSpeechRecognition` API for real-time interim + final transcripts.
- Fallback: `MediaRecorder` → WebM/Opus audio blob → backend `/api/interview/transcribe/` → Whisper (via Groq `whisper-large-v3`) for higher accuracy.
- Session transcript buffer accumulates all prior Q&A pairs.

**2. Dynamic Question Generation**

The LLM receives the full conversation history and generates the next question based on:

- **Prior context**: all previous questions and the candidate's answers.
- **New data**: the just-received answer, particularly any claims, gaps, or interesting points.
- **Session configuration**: target role, company, uploaded JD keywords, and difficulty level.

Prompt structure:
```
System: You are a senior {role} interviewer at {company}. Interview the candidate for {n} questions covering technical depth and behavioral competency. Adapt follow-up questions based on the candidate's prior answers.

Conversation history:
Q1: ...  A1: ...
Q2: ...  A2: ...
[current answer]

Generate the next interview question. If the candidate mentioned {X}, explore it deeper.
```

This enables genuinely adaptive interviews — if a candidate mentions Kafka in a system design answer, the LLM can probe their Kafka knowledge in the next question.

**3. Answer Evaluation**

After the final question:
- All transcripts are sent to the LLM for holistic evaluation.
- Scoring dimensions: technical accuracy, communication clarity, problem-solving approach, relevant examples.
- Per-question scores + overall rating returned on the results page.

**4. Text-to-Speech (TTS) Output**

- Browser-native: `window.speechSynthesis` for low-latency playback.
- Optional backend TTS: Groq / ElevenLabs for higher-quality voices.

**5. Planned API Endpoints**

| Endpoint | Method | Description |
|---|---|---|
| `/api/interview/start/` | POST | Create interview session, return `session_id` |
| `/api/interview/transcribe/` | POST | Whisper transcription for audio fallback |
| `/api/interview/respond/` | POST | Send transcript + history → receive next question |
| `/api/interview/evaluate/` | POST | Full transcript → per-question scores + verdict |
| `/api/interview/stop/` | POST | Finalize session |

**6. Planned `InterviewSession` Model**

Open `http://localhost:5173` in your browser.

**Application flow:**
1. Register / Log in.
2. Upload your resume and (optionally) a job description on the Dashboard.
3. Click **Start Mock Test** → choose Standard or Custom mode → begin the proctored test.
4. Review your results, proctoring report, and per-question breakdown.
5. Click **Start AI Interview** → upload resume → enter company/role → begin the interview.
6. Check your session history on the Dashboard for streak and score stats.
